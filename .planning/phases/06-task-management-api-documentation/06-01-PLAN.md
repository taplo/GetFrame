    **`stop_pipeline(&self, id: &StreamId) -> bool`:**
    - Remove the pipeline handle from `self.pipelines` (lock, remove, then drop lock).
    - Send `PipelineExitReason::UserInitiated` via `exit_tx` to signal the reconnection task to stop.
    - Cancel the `shutdown_token`.
    - Join the decode thread if `join_handle` is Some.
    - Decrement `STREAMS_ACTIVE` gauge.
    - Return `true` if a pipeline was removed, `false` if `id` was not found in the pipelines map.
    - Do NOT remove the entry from `self.registry` — this is the key difference from `remove_stream`.
    - Log at info level: `"Pipeline stopped for stream {id}"`.

    **`start_pipeline(&self, id: &StreamId) -> bool`:**
    - Look up stream info from `self.registry.get(id)`. Return `false` if not found.

    - Spawn `self.spawn_frame_consumer(...)` for the new pipeline.
    - Create `PipelineHandle` and insert into `self.pipelines`.
    - Spawn `self.spawn_reconnection_task(...)` with the new handle's exit_rx.
    - Increment `STREAMS_ACTIVE`.
    - Return `true`.

    State machine transitions enforced in `start_task`/`pause_task`/`resume_task`/`stop_task`:
    - `start_task`: Only if status is `Created` or `Error`. Calls `stream_manager.start_pipeline(...)`. Sets status to `Running`, sets `started_at` to now.
    - `pause_task`: Only if status is `Running`. Calls `stream_manager.stop_pipeline(...)`. Sets status to `Paused`.
    - `resume_task`: Only if status is `Paused` or `Error`. Calls `stream_manager.start_pipeline(...)`. Sets status to `Running`.
    - `stop_task`: Only if status is `Running` or `Paused`. Calls `stream_manager.stop_pipeline(...)`. Sets status to `Stopped`, sets `stopped_at` to now.
    - `delete_task`: Any status allowed. If task was Running, calls `stream_manager.stop_pipeline(...)` first. Returns true if task existed.

    `TaskError` enum (derive `Debug, thiserror::Error`):
    - `#[error("Invalid state transition: {0}")] InvalidTransition(String)`
    - `NotFound` / `#[error("Task not found")] NotFound`
    - `#[error("Internal error: {0}")] Internal(String)`

    When starting a pipeline for a task (`start_task`/`resume_task`):
    - First add the stream config to StreamRegistry via `registry().add()`? NO — look at the code flow more carefully.
    - Actually, `StreamManager::start_pipeline(id)` assumes the stream already exists in `StreamRegistry`. So for a task, we need to first register the stream config in `StreamRegistry` before calling `start_pipeline`.
    - For now, add the stream to registry via `stream_manager.add_stream(config)` which both registers AND starts the pipeline. Then we'd need to... hmm.
    - Better approach: For a task's start/resume, generate a new `StreamId`, register it in the stream registry with the stream config, then call `stream_manager.start_pipeline(new_id, ...)`.
    - Wait, but `start_pipeline` takes a `StreamId`, not a config. It expects the config to already be in the registry.
    - So the flow is: generate StreamId → add to registry → call start_pipeline(id).
    - But `add_stream` does both register AND start. We need just "register" without starting.
    - So actually we need to call `registry().add(id, config)` directly then `start_pipeline(id)`.
    - This is fine because `StreamManager::registry()` returns a public `&StreamRegistry`.

    Revised flow for `start_task`:
    1. Validate current status allows start
    2. Generate new `StreamId`
    3. Register stream config in registry via `stream_manager.registry().add(stream_id, config)`
    4. Call `stream_manager.start_pipeline(&stream_id)`
    5. If start_pipeline fails, remove from registry and return error
    6. Update task status to Running, set stream_id and started_at
    7. Return updated TaskInfo

    Similarly for `resume_task` — same flow (generates new StreamId each time; pause already stopped + removed the old pipeline handle).

    For `pause_task` and `stop_task`:
    1. Get the task's `stream_id`
    2. Call `stream_manager.stop_pipeline(&stream_id)` which removes pipeline handle but keeps registry entry
    3. IMPORTANT: Also remove from registry via `stream_manager.registry().remove(&stream_id)` to clean up
    4. Update task status (clear stream_id)
    - Wait, stop_pipeline keeps registry entry. But we want to keep the StreamConfig for potential resume. So we should:
      - Stop the pipeline (removes pipeline handle from pipelines map, keep registry entry)
      - Mark status as Paused/Stopped, keep stream_id... but if we resume we need a new StreamId anyway.
      - Since each start/resume generates a new StreamId, we should also clean the old one from registry.
      - So: stop_pipeline(stream_id) → registry.remove(stream_id) → clear stream_id in task info.

    For `stop_task`: same flow. Stop pipeline, remove from registry, mark Stopped.

    For `delete_task`: If Running or Paused, stop pipeline + remove from registry. Remove task from TaskRegistry.
